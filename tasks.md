Task 3 [DONE]
The controller agent always runs infinitely until it emits the STOP command which is fine. But this could mean that after a while this could consume all codex and claude code tokens from our subscription / ie.: it could exceed the allowance in that period then it would just keep erroring (like a rate limit error etc). Now we need a way to solve this.
I think a simple one could be just a setting that would wait before starting the next step (when the agent says "Thinking about next step" or something).
By default it would just keep running instantly like it does now, but we could change a setting such that it waits between controller runs (so like before we launch the codex command each time or something we would wait the specified time). We would be able to set things like 1, 2, 3, 5, 10, 15, 30 minutes, 1, 2, 3, 5, 6, 12 hours, 1, 2, 3, 4, 5, 6, 7 days (so 7 days would run weekly which is the maximum wait time). As seen we would allow very long wait times so we would need quite an efficient way to do this (not something that consumes CPU or something but something async if you see what I mean). Now the tricky part is what if the user reloads the VSCode screen or reloads the terminal (this should work in the terminal / cli version too). Somehow we should be able to just restore the timer where it left off. But yeah maybe we could do time based self activation or something (so like we set the agent to sleep and activate in 5 minutes by setting it to activate at like 10:05 (if the current time is 10:00)). Then no matter what we do / even if we reload the window it will still know when it will need to reactivate (it just reads the current timer settings, it goes back to auto sleep and auto wake up at the specified time). Also we need to be careful not to auto wake up the agent if it's already woken up / running or maybe the user already started it manually by sending it a message so this can be tricky to implement. Well maybe it's not that difficult because the auto sleep and wake up would be only set if the agent is invoking itself automatically (ie it is not emitting the STOP command so it will continue running). So it's like a setTimeout behavior in JavaScript. If the agent is run manually by the user after a setTimeout has already been set (to wake up the controller / ie.: to run the intended codex command) then the setTimeout should simply just be cleared, that's it. Then the next time it wants to invoke the codex command we wait again before doing so (if the user sends a message we always want to run codex instantly, even if there is a wait setting being set up, like to wait 5 minutes between runs). Now ideally this setting should be configurable via a slash command (on both the terminal and also on the VSCode extension) and on the VSCode extension there should also be a dropdown in the chat input box (like the controller model dropdown) where we could set the wait time between controller auto invocations (by default it is set to none which is exactly the behavior we have currently)
> Completed: Added durable per-run wait scheduling with shell and VSCode controls, reload-safe wake restoration, and regression coverage in tests/wait-delay.test.js.

Note that we should only do this with genuine errors, not if the user interrupts the session (which also raises an error from the clis, but we already catch that error though and just show "Run stopped by user." so yeah that's fine). So yeah this feature builds on top of task 3 really (similar auto wait feature but not config based). The edge case is if there is already a wait time specified between codex invocations. Well maybe if there is an error we should always just wait a fixed 30 minutes and ignore the config (so for example the wait time config is set to be 1 day so it would run daily. An error happened with the invocation of codex (for example caused by a rate limit error or something), then instead of waiting for an entire day to try again, we would only wait for 30 minutes because technically the run for today did not finish successfully. If successful then we wait for a day as per the config normally)
